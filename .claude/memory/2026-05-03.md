# Session Summary — 2026-05-03

## コンテキスト

長尺曲分析時の OOM Kill 問題を起点とした壁打ちセッション。原因究明（実装ミスか環境制約か）→ 修正（PR #30 マージ）→ 提供アーカイブ `ugh_music_project_lite` の評価 → 移植前検証（DeltaECalculator は移植不可と判定）→ 自前の dynamics 記述子実装（PR #31 マージ）まで一気通貫。「設計→実装→セルフレビュー→PR」を Claude が直接担当する特例セッション（ユーザーが明示許可）。

## 修正・訂正

- **`audio_loader.py` の二重 load (PR #30)**: ステレオ曲で `librosa.load(mono=True)` と `librosa.load(mono=False)` を 2 回呼んでいた。30s 入力でメモリ +173MB（理論値の 33 倍）→ 90s で OOM Kill。1 回 `mono=False` で load して mono は `np.mean(axis=0)` で派生する形に統合
- **「セグメント間エネルギー変化率」の概念整理**: アーカイブ `DeltaECalculator` は「新発明」ではなく標準 MIR の novelty curve の劣化版（RMS のみ + 均等分割 + 0.85 下限クランプで飽和）。現リポジトリには既に `compute_novelty_curve` という上位互換実装が `structure_novelty.py` に存在することを発見、移植不要と再判定

## 成功パターン

- **「実装ミス vs 環境制約」の切り分けを計測で確定**: ユーザーが psutil で per-step プロファイリングを取得（30s で `librosa.load` 単独が +173MB / 全体の 74%）。これにより仮説段階の「pyin が主犯」「resample_poly が主犯」を一掃し、二重 load という設計バグに収束。憶測ベースの打ち手より計測駆動が圧倒的に速い
- **「便宜的か、確立された概念か」を文献軸で判定**: アーカイブの ΔE 計算式について、ユーザーから「既存ツールで判断できる概念か」と問われた際、Foote (2000) novelty / EBU R128 LRA / Spectral flux 等の標準 MIR 概念と対比して「概念は標準・実装は劣化版」と整理。これが「移植せず自前実装」の判断根拠になった
- **検証フェーズを実装前に挟んだ**: ユーザーから「移植する前に定義と計算が本当に正しいのか検証する必要がある」と指示があり、Explore agent で 6 軸調査（数学的妥当性 / 用語衝突 / 閾値根拠 / 飽和 / 分割戦略 / 経験的検証）を実施。結果として「ΔE」名称衝突と 0.85 下限クランプによる構造的飽和を発見、移植判定を「部分移植可」から「移植不可」に修正
- **`_db` suffix で命名衝突を予防**: 提供アーカイブの「ΔE」が現リポジトリの `DeltaEProfile.transition_type` と同名別概念だった反省から、新指標は `dynamic_range_db` と suffix 明示。docstring でも「NOT EBU R128 LRA」と明記、後続が混同しない
- **既存 novelty curve の再利用で計算コストゼロ追加**: `_detect_sections_v2` 内部で計算済みの novelty curve を tuple 返却に変更して `extract_physical` で再利用。新指標を 2 つ追加してもメモリ問題への影響なし
- **snapshot drift を regenerate スクリプトで対応**: PhysicalRPE 拡張で 5 件の snapshot test が drift。エラーメッセージ通り `python scripts/regenerate_expected.py` で再生成、新フィールドが正しく入っていることを diff で確認

## 設計判断

- **アーカイブの DeltaECalculator は移植せず、概念だけを再実装**: 「セグメント間エネルギー変化率」というアイデアは MIR 標準だが、アーカイブ実装は (a) RMS only / (b) 均等分割（structure 境界無視）/ (c) 0.85 下限クランプで飽和 / (d) 閾値の統計的根拠なし、と問題が複数。現リポジトリの `compute_novelty_curve`（RMS + onset + spectral flux + chroma の重み付き複合）を再活用する方が筋が良い
- **`dynamics_summary` の 5 フィールド設計**: peak / mean / std / event_count / temporal_balance。最後の `temporal_balance` は first-half mean / whole mean で、>1 = 前半重い、<1 = 後半重い。曲のダイナミクス構造を最小限の数値で表現。`event_count` の閾値 (mean + 0.5*std) は既存 `find_boundaries` と一貫
- **`dynamic_range_db` は EBU R128 LRA ではない**: proper LRA は K-weighted short-term loudness + 二段ゲーティングが必要で実装コストが大きい。本指標は P95/P10 frame RMS の dB 比という軽量プロキシ。`_db` suffix と docstring で明示的に区別
- **新フィールドはすべて Optional**: `_detect_sections_v2` が短尺音源（≤5s）で novelty curve を計算しない既存 short-circuit に従い、`dynamics_summary` も None を返す。後方互換性を保ったまま機能追加
- **「良い曲」指標 → 作曲の願望はリポジトリ射程に入れない判断保留**: ユーザーの長期願望は normative（規範的）レイヤー。本セッションでは descriptive（記述的）レイヤー強化に集中。ただし descriptive を厚くしておけば後から normative を乗せられる構造を維持
- **メモリ問題はローカル PC 実行で実用上解決**: 現環境（Claude Code sandbox ≈ 800MB-1GB）では 90s 超で OOM だが、3-5 分の通常曲なら 16GB PC で約 1.6-2.6GB peak で完全に収まる。Fix #30 で peak が更に半減するので 16GB PC なら 30 分曲まで余裕。1 時間級の長尺対応はセクション単位ストリーミング処理が必要だが、当面の射程外

# Session Summary — 2026-05-03 (Session 2: AI 音楽生成ブレインストーミング)

## コンテキスト

## 議論の到達点

- **AI 生成曲の問題は学習分布の平均回帰**であり、これは音楽的特徴ではなく生成モデルの本質的性質の多面的発現
- **チャート選別 ≠ 時代越え選別**。クラシックは生存バイアスの結果、当時の "外れ値" だけが残った
- **裾の survivor を狙う問題は、抽出問題に還元される**。意味ベクトル空間の "右上座標"（coherent × 端的）にアクセスする手段が必要
- **クラシックが残った理由は「楽譜という時代独立の抽象層」**。AI 音楽でも楽譜（決定論）と演奏（確率）を分離できれば原理的に同じ構造が成立する
- **使われる音楽 vs 消費される音楽**: 民謡・民族音楽の survivor 性の核は「共同体に道具として供給される機能」。著者性は survivor 性の必要条件ではない（民謡が反証）
- **評価関数の Goodhart 問題**: 単一評価関数最大化は平均回帰を再発させる。目的多様性が survivor 性の真の必要条件
- **「秩序ある破壊と創造」**: 平均でも noise でもない coherent な edge を、定義 + 状態として AI に渡す必要がある
- **画像分野からの transfer**: ControlNet 等で「構造化仕様 + AI 生成器」アプローチの原理的妥当性は既に検証済み。残るは音楽特有の粒度設計（工学問題）

## 設計判断

- **拡張検証トラックとして別立て**: AI 音楽 DAW ビジョンは野心的なため、本流ロードマップ（roadmap.md / roadmap_goal1.md）の本丸に組み込まず、`docs/ai_music_daw_vision.md` として独立配置。本流の進捗を阻害しない設計
- **PoC (1) を Q0 に統合**: 「survivor 抽出妥当性検証」は Q0-1 のサンプル選定で並行達成（クラシック / 民謡 / 民族音楽の CC0 音源を含めるだけ）。追加工数ほぼゼロ
- **戦略は "ツール" ではなく "標準"**: SVP を「AI 音楽の MIDI」相当のオープン標準として位置付ける。単独でツール開発するより leverage が大きい。MIDI のアナロジー
- **(2)-(5) PoC は (1) の結果待ち**: rarity / coherence / 合成・再演 / 評価ループは投資コスト大、(1) で前提が validated されるまで保留
- **完全再現ではなく semantic 一致を default 基準に**: SVP→SVP は strict、SVP→audio は semantic（複数 rendering が "同じ作品" なら accept）。クラシックの「楽譜は厳密、演奏は解釈」と同型
