## 設計判断

- **マージ済みブランチの削除は Codex に委ねた**: Claude のリサーチ結果（重複 7 本のリスト）を Codex に渡し、削除実行は Codex が担当。Claude は「コードを書かない」原則を保ちつつ、調査価値を最大化
- **`compare` から `--separate` 完全削除（A 案）**: docstring 注記（C 案）も検討したが、運用上の混乱を避けるため flag 自体を削除。`run --separate` / `evaluate --separate` で per-stem 分析できるので機能性は保たれる
- **`STEM_BASELINE_PROFILES` の根拠コメント**: `vocals=acoustic / drums=edm / bass=edm / other=pro` の選択理由（spectral 特性ベース）と「ジャンル指定したい場合は parent baseline を上書き」の運用方針をコード上にコメント化。レビュアーが将来納得しやすい状態
- **CLI 重複オプションを `Annotated` で集約**: `extract` / `evaluate` / `run` / `batch` の 4 コマンドで `--separate` / `--separation-model` / `--separation-device` を共通化。`Annotated[bool, typer.Option(...)]` 型エイリアス 3 つで -41 行の減量
- **`RPEScore.stem_scores` の再帰防止 validator**: 現状の `score_rpe()` は flat 出力なので問題ないが、将来誰かが直接 `RPEScore(stem_scores={...nested...})` で構築するのを `field_validator` で防御。テストで pin 済み
- **stem reconstruction validation の `max` への zero-pad**: `min` ベースの overlap-only 比較を捨て、`max(source, recon)` への zero-pad で union 全域を residual 計算。tail loss が silent drop されない。`length_delta_samples > allowed_delta` の追加 fail-gate は不要と判断（padding で十分カバー、source 側に音があれば必ず residual に反映）
- **`scripts/validate_stem_separation.py` を CI 不参加の sign-off ツールとして配置**: Demucs は重く CI で常時実行は非現実的。CI は `tests/test_stem_validation.py` の合成データテストで決定論的に検証、real-audio 検証は手動 sign-off に分離
